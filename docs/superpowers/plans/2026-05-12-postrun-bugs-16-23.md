## Bug 16 — PM proceeds on empty Phase 2 reports (DEGRADED gating)

**Symptom:** Bear Researcher (UI) and Bull Researcher (CAMT) both returned the fixed-length `[Error in Bear Researcher: ...]` placeholder (~72 chars) after a ReadTimeout where the retry was skipped because the wall-clock budget would have expired. The QC service flagged it with `[WARNING] Report section 'bear' is suspiciously short (72 chars) for UI. Flagging input.` and prepended `[SHORT INPUT DETECTED: Length 72 < 200]` to the report — but the PM call ran anyway, producing UI=AVOID/LOW with no bear view and CAMT=WATCH/MODERATE with no bull view.

**Root cause:**

- `app/services/research_service.py:631-636` dispatches Bull/Bear/Risk via `ThreadPoolExecutor(max_workers=6)`.
- `app/services/research_service.py:1529` emits `Skipping {wait}s backoff; wall-clock budget would expire` — retry is skipped, exception bubbles up.
- The `run_agent` wrapper at `app/services/research_service.py:~628` catches the exception and returns `f"[Error in {name}: {e}]"` (a fixed-format short string).
- `app/services/quality_control_service.py:18-22` only **annotates** short reports — it never blocks the PM.
- The PM is called at `app/services/research_service.py:684-686` regardless.

**Fix:**

1. In `app/services/quality_control_service.py`, add a second classifier alongside `validate_reports` that returns a structured result:
   ```python
   @dataclass
   class ReportQuality:
       short_keys: list[str]           # below 200 chars
       error_keys: list[str]           # text starts with "[Error in "
       missing_keys: list[str]         # None
       degraded: bool                  # any of: critical key short/error/missing

       @property
       def critical_degraded(self) -> bool:
           # Phase 2 keys are "critical" — if any of bull/bear/risk is empty, the PM is uninformed.
           return bool(set(self.short_keys + self.error_keys + self.missing_keys) & {"bull", "bear", "risk"})
   ```
   Keep `validate_reports` (backward compat) and add `assess_quality(reports, ticker, critical_keys) -> ReportQuality`.

2. In `research_service.py`, after Phase 2 completes and before the PM call, call `assess_quality` for `{"bull", "bear", "risk"}`. If `critical_degraded` is true:
   - **Do not skip the PM** (we still want a verdict) — but pass a `degraded_phase2=True` flag into the PM prompt asking it to mark `conviction="LOW_DEGRADED"` and add a `data_quality_warning` field to the JSON.
   - Persist that flag to `decision_points` via a new column `phase2_degraded INTEGER DEFAULT 0` (migration in `init_db`). The dashboard view should render a "⚠ Phase 2 degraded — PM ran on incomplete debate" banner on the row.
   - Tag the trade-report CSV row with `phase2_degraded=1` so downstream readers can filter.

3. Add unit test `tests/test_pm_degraded_gating.py` that constructs a state with `bear="[Error in Bear Researcher: ReadTimeout]"` and asserts:
   - `assess_quality` reports `critical_degraded == True`.
   - The PM prompt-builder includes the degraded notice (search for a sentinel substring).
   - The persisted row has `phase2_degraded=1`.

**Why not refuse to run the PM:** the user explicitly considered that as "at minimum" the row should be DEGRADED-tagged. Refusing entirely loses the partial signal and breaks the daily cohort. DEGRADED-tag is the right balance — keep the row, mark it.

**Files changed:**

- `app/services/research_service.py` (~684, gate the PM call's prompt + persist flag)
- `app/database.py` (add `phase2_degraded` column + migration)

## Bug 17 — Stop-guard downside ceiling (abort instead of widening past 20%)

**Symptom:** FORM had its stop widened from $104.53 → $68.77 via `widened_to_sma_200`, producing upside +7.7% / downside −41.0% / R:R 0.2. Same shape (smaller magnitude) for `widened_to_2x_atr` on UI, ASTS, CAMT, CRWV, HIMS.

**Root cause:** `app/utils/stop_loss_guard.py:29-84` widens whenever the PM's stop violates `entry_low - 1.5 * ATR`, picking the **farther** of `entry_low - 2*ATR` and the nearest sub-entry SMA. There is no upper bound on resulting downside %. `recompute_risk_metrics` (lines 87–110) computes the downside but does not feed back into a reject decision.

**Fix:**

1. Add an `abort_threshold_pct` parameter to `widen_stop_if_too_tight` (default 20.0). After computing `new_stop` (lines 73–82), compute candidate downside:
   ```python
   candidate_downside = (entry_low - new_stop) / entry_low * 100.0
   if candidate_downside > abort_threshold_pct:
       return StopLossAdjustment(stop_loss=None, adjusted=False,
                                  reason=f"abort_downside_{candidate_downside:.0f}pct")
   ```
2. In `research_service.py` (or wherever the guard is invoked — likely near the PM post-check), interpret `reason.startswith("abort_")` as "downgrade the recommendation to WAIT_FOR_STAB and add a `risk_too_wide=True` flag to `decision_points`." Do **not** silently pass through — the user explicitly said the guard should refuse the trade.

3. Persist `stop_guard_reason TEXT` (already exists? grep first; if not, add column) so downstream readers can audit why a row went to WAIT.

4. Tests in `tests/test_stop_loss_guard.py` (add or extend): the FORM scenario must return `adjusted=False, reason="abort_downside_41pct"`. Existing widening tests should pass with the default ceiling.

**Configuration:** make the threshold an env var `STOCKDROP_STOP_DOWNSIDE_CEILING_PCT` (default 20.0) so we can tune without redeploy.

**Files changed:**

**Acceptance:** replay FORM 2026-05-11 fixture → recommendation is WAIT_FOR_STAB (not BUY/BUY_LIMIT), `stop_guard_reason="abort_downside_41pct"`.

## Bug 19 — Finnhub per-ticker circuit breaker

**Symptom:** RDY hit four separate 10-second Finnhub timeouts in sequence (`company_news`, `filings`, `company_earnings` ×2). Cost: ~40s wall time for one ticker that returned nothing.

**Root cause:** `app/services/finnhub_service.py:33-47` retries each call once with a 2s backoff. There is no per-ticker breaker — once one endpoint times out, the next endpoint for the same ticker has no signal to short-circuit. Call sites (`stock_service.py:1091`, `1258`, and `finnhub_service.py:166`) call them sequentially.

**Fix:**

1. Add a module-level breaker dict in `finnhub_service.py`:
   ```python
   _breaker: dict[str, float] = {}  # symbol -> unix-ts when block expires
   _BREAKER_WINDOW_SEC = 60          # block this ticker for 60s after any timeout
   _BREAKER_TRIGGER_FAILURES = 1     # one failure is enough — Finnhub timeouts cluster
   ```
2. Wrap each public method (`get_company_news`, `get_filings`, `get_earnings_facts`, `get_latest_reported_quarter`):
   ```python
   def _breaker_open(symbol: str) -> bool:
       exp = _breaker.get(symbol)
       return exp is not None and time.time() < exp

   def _trip_breaker(symbol: str):
       _breaker[symbol] = time.time() + _BREAKER_WINDOW_SEC
       logger.info(f"[Finnhub] tripping breaker for {symbol} for {_BREAKER_WINDOW_SEC}s")
   ```
3. At the top of each method, check `_breaker_open(symbol)` → return `[]` / `{}` immediately. In the `except` branch of `_call_with_retry`, if the exception is transient (timeout / 5xx), call `_trip_breaker(symbol)` before re-raising.
4. Add a `reset_breaker(symbol=None)` test helper.

**Tests:** `tests/test_finnhub_breaker.py` — simulate a `requests.Timeout` on `get_company_news("RDY", ...)`, then call `get_filings("RDY", ...)` and assert it returned `[]` without making an HTTP call (mock `self.client.filings` and assert call_count == 0).

**Files changed:**
- `app/services/finnhub_service.py`
- `tests/test_finnhub_breaker.py` (new)

**Acceptance:** RDY-style 4-timeout scenario costs 1 timeout (≤12s) instead of 4 (~40s).

## Bug 20 — Bollinger gatekeeper uses non-split-adjusted prices

**Symptom:** VCX (price $239.52 / lower_bb $10.51), SITM ($847.65 / $342.51), SNDK ($1456.13 / $688.03), MXL ($94.61 / $7.93). All passed `%B` at the top of the band and got rejected as "Not Dip Enough" — correct outcome, but the underlying numbers indicate the 20-day BB window straddles a stock split and is using unadjusted history.

**Root cause:** `app/services/tradingview_service.py:427-432` fetches via TradingView `TA_Handler` with no split-adjust kwarg. The `tvDatafeed`/TA_Handler API does not expose a split-adjustment toggle — TradingView's intraday BB indicator computes over the raw price stream visible on the chart. For a recent split, this leaves stale pre-split prices in the 20-day window.

**Fix — option A (preferred, deterministic):** compute the BB ourselves from yfinance split-adjusted closes when the gatekeeper would otherwise reject as "Not Dip Enough" *and* the divergence ratio is suspicious.

Concretely, in `app/services/gatekeeper_service.py` around the BB fetch (lines 121–123):

1. After fetching `bb_lower`, `bb_upper`, `current_price` from TradingView, compute `divergence = current_price / bb_lower if bb_lower else None`.
2. If `divergence is not None and divergence > 3.0` (heuristic: healthy band is within ~1.5× of price; >3× is almost certainly a split), recompute the band from yfinance:
   ```python
   import yfinance as yf
   hist = yf.Ticker(symbol).history(period="2mo", auto_adjust=True)["Close"]
   # auto_adjust=True applies split + dividend adjustments
   if len(hist) >= 20:
       sma20 = hist.rolling(20).mean().iloc[-1]
       std20 = hist.rolling(20).std().iloc[-1]
       bb_lower_adj = sma20 - 2 * std20
       bb_upper_adj = sma20 + 2 * std20
       # Use adjusted values; log the swap.
       print(f"[Gatekeeper] {symbol}: BB swap (raw_lower={bb_lower:.2f} -> adj_lower={bb_lower_adj:.2f}); detected split in window")
       bb_lower, bb_upper = bb_lower_adj, bb_upper_adj
   ```
3. Continue with the existing %B calculation.

**Fix — option B (cheap, partial):** flag rather than fix — emit a `[BB-SPLIT-SUSPECT] {symbol}` warning and exclude the ticker from screening entirely (treat as if `%B` were NaN). This is one-line and removes the bad data from the cohort but loses the ticker. Reasonable if a recent split is the only cause.

**Recommendation:** A. It's ~10 LOC, deterministic, and matches the project's "trust adjusted data" preference (yfinance with `auto_adjust=True` is the canonical source for split-adjusted equities in this repo).

**Tests:** mock TradingView returning `bb_lower=$10` for a $239 ticker; mock yfinance returning a clean 2-month adjusted history; assert the swapped lower-bb is within 0.5× of the current price.

**Files changed:**
- `app/services/gatekeeper_service.py`
- `tests/test_gatekeeper_split_swap.py` (new)

**Acceptance:** replay VCX/SITM/SNDK/MXL 2026-05-11 fixtures → the logged `bb_lower` after swap is within 30% of current price (not 10–25× off). Gatekeeper decision (PASS/REJECT) can still be "Not Dip Enough" — we're fixing the **numbers**, not the verdict.
